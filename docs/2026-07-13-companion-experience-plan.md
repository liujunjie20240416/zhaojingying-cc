## 1. 私密关系时间线

### 数据边界

时间线必须属于 `Friend`，而不是公共 `Character`。同一个公共角色可能被不同用户使用，任何导入的私密关系都不能展示给其他用户。

建议新增：

- `RelationshipMilestone`：类型、标题、摘要、日期、日期精度、来源、确认状态、可见状态。
- `MilestoneEvidence`：关联导入消息 `msg_index` 或在线消息 `Message.id`。
- 类型：初次认识、关系阶段、纪念日、第一次旅行、重要约定、争执与和好、用户确认的共同回忆。

### 生成流程

1. 从日级摘要和关系事实中提取“候选里程碑”。
2. 代码归一化日期并做精确去重，LLM 只负责语义合并。
3. 模糊日期、争执、关系阶段变化默认要求用户确认。
4. Reflection 只提出在线聊天产生的新候选，不直接发布。
5. 回答时只检索当前问题相关的里程碑，不把整条时间线塞进系统提示词。

### 前端

- 聊天顶部增加“我们的时间线”入口。
- 手机端使用全屏纵向时间线；桌面端使用侧边抽屉。
- 每项可查看证据、确认、编辑、隐藏；默认只有当前用户可见。

推荐默认：自动提议、用户确认后展示；“争执与和好”默认隐藏，可在设置中开启。

## 2. 可控的主动陪伴

### 推荐 MVP

- 首期只做：用户创建的提醒、纪念日、连续 7 天未聊天后的自然问候。
- 只发站内消息；推送通知放到下一阶段。
- 默认关闭，用户主动开启。
- 最多每天 1 条；默认免打扰 22:30–08:00；支持暂停和完全关闭。
- 文案禁止内疚诱导或假装实时在线，例如禁止“你怎么还不理我”。

### 模型

- `CompanionSettings`：总开关、事件类型开关、频率、时区、免打扰时间、每日上限。
- `ProactiveJob`：触发原因、计划时间、状态、去重键、实际发送时间。
- 调度器只创建持久任务；Worker 抢占任务、生成消息并写入普通在线聊天记录。

### 前端

- 好友设置增加“主动陪伴”。
- 提供低/标准/高三个频率档和高级设置。
- 每条主动消息标记触发原因，可一键关闭该类型。
- UI 使用“陪伴消息”，不显示“真人在线”含义。

## 3. 图片多模态

### 消息链路

1. 前端选择图片、预览、压缩并上传。
2. 服务端校验 MIME、大小和像素，移除 EXIF；原图/缩略图放媒体存储，不把 Base64 存数据库。
3. 新增 `MessageAttachment`，保存消息归属、存储地址、尺寸、类型、OCR/描述等派生信息。
4. `VisionModelAdapter` 把文字和图片组织成模型所需的多模态输入；只有含图片时才路由到视觉模型。
5. 图片描述/OCR 可进入统一历史检索，但原图默认不进入长期记忆；需由 Reflection 提取并经现有记忆规则处理。

### API

- 独立上传接口返回 `attachment_id`。
- 聊天接口从纯 `message` 扩展为 `message + attachment_ids`。
- 当前供应商是否支持视觉能力由模型适配器和配置检查决定，业务层不绑定某个模型。

## 4. Harness（AI 工程护栏）

Harness 不是一个用户可见功能，而是一套让 Agent 改动可测试、可回放、可度量的工程系统：

- 数据集：普通闲聊、回忆问题、冲突事实、情绪问题、图片问题、风格案例。
- 断言：路由是否正确、检索来源、证据是否存在、回答是否越权、气泡格式、风格长度。
- 指标：Recall/Hit@K、事实一致性、无依据编造率、风格相似度、延迟、Token 和成本。
- 运行方式：单元测试用假模型；集成测试使用固定录制响应；少量真实模型评测独立运行，不混入离线 CI。
- 可观测性：Trace 做隐私脱敏，并支持从失败 Trace 一键生成回归案例。

优先级高于 MCP、多 Agent：先有 Harness，后续增加工具或 Agent 才能知道系统究竟变好还是变差。

## 5. ImportRun、IndexJob 与 Checkpoint 的区别

- `ImportRun`：一次导入尝试的总账，记录文件指纹、版本、状态和是否成为当前激活版本。解决导入历史、原子切换和回滚。
- `PreprocessingCheckpoint`：一次导入内部每个 Map Chunk 的已完成结果。本轮已实现，解决进程中断后从完成位置继续。
- `IndexJob`：把 SQLite/PostgreSQL 中的权威记录投影到 LanceDB/pgvector 的持久任务。解决向量索引失败、重试和重建。

可以类比为：`ImportRun` 是订单，Checkpoint 是订单里的每个处理步骤，`IndexJob` 是之后的仓库上架任务。
